    **ou_model.py** (QTIM-01, QTIM-03):
    - `estimate_ou_parameters(prices: np.ndarray, dt: float = 1.0) -> tuple[float, float, float, float]`
      Exact implementation from Research code example:
      - dX = kappa * (theta - X) * dt + sigma * dW
      - OLS estimation: dx = a + b*x + noise where a = kappa*theta*dt, b = -kappa*dt
      - Compute kappa = -b/dt, theta = a/(kappa*dt) if kappa > 0 else mean(prices)
      - sigma = std(residuals) / sqrt(dt)
      - confidence = R-squared of regression, clamped to [0, 1]
      - Return (0.0, mean(prices), 0.0, 0.0) if len < 30 or denominator near zero

    - `compute_entry_window(kappa: float, theta: float, sigma: float, current_price: float, confidence: float) -> tuple[float, float, float]`
      - If kappa <= 0 or confidence < 0.1: return (0.0, 0.0, 0.0) -- no mean-reversion detected
      - displacement = theta - current_price
      - direction = +1.0 if displacement > 0 (price below mean, expect rise) else -1.0 (price above mean, expect fall)
      - half_life = np.log(2) / kappa if kappa > 0 else float('inf') -- time to revert halfway
      - urgency = min(1.0, abs(displacement) / (2 * sigma + 1e-10)) -- how far from mean in sigma units
      - If half_life < 5 bars: timing is imminent, boost urgency by 1.5x (capped at 1.0)
      - window_confidence = confidence * min(1.0, urgency) -- low urgency near mean = low timing confidence
      - Return (direction, urgency, window_confidence)

    - `compute_exit_window(kappa: float, theta: float, current_price: float, entry_price: float) -> tuple[float, float]`
      - Estimate time to mean reversion: half_life = ln(2)/kappa if kappa > 0
      - If current_price is moving away from theta (wrong direction), return (exit_urgency=1.0, 1.0) -- exit now
      - If profit target within half_life bars, return (exit_urgency=0.3, 0.8) -- hold
      - Return (exit_urgency 0-1, confidence)

    **phase_transition.py** (QTIM-02):
    - `compute_atr(high: np.ndarray, low: np.ndarray, close: np.ndarray, period: int = 14) -> np.ndarray`
      Exact implementation from Research code example:
      - True Range = max(high-low, abs(high-prev_close), abs(low-prev_close))
      - Wilder smoothing: ATR[i] = ATR[i-1] * (period-1)/period + TR[i]/period
      - First `period` values use simple average of TR
      - Handle edge case: if len < period, return array of simple average TR

    - `detect_phase_transition(close: np.ndarray, high: np.ndarray, low: np.ndarray, atr_period: int = 14, compression_threshold: float = 0.5, expansion_threshold: float = 2.0) -> tuple[str, float, float]`
      - Compute ATR with compute_atr()
      - Compute long-term ATR average: mean of ATR over full series
      - Current ATR ratio = ATR[-1] / long_term_avg
      - If ratio < compression_threshold: state = "compression", energy = 1.0 - ratio (stored energy, breakout imminent)
      - If ratio > expansion_threshold: state = "expansion", energy = ratio - 1.0 (energy releasing)
      - Else: state = "normal", energy = 0.0
      - Confidence = min(1.0, len(close) / (3 * atr_period)) -- need enough data for meaningful ATR
      - Return (state, energy, confidence)

      The "energy" concept: volatility compression stores energy (like a coiled spring). When compression breaks, it releases as a directional move. The direction comes from other modules; timing tells the fusion core WHEN the move is likely.

    **module.py**:
    - `class QuantumTimingModule` implementing SignalModule Protocol
    - Constructor takes `config: TimingConfig`
    - `name` property returns `"timing"`
    - `async def initialize(self) -> None`: no-op (no warm-up needed for scipy)
    - `async def update(self, tick_arrays, bar_arrays, dom) -> SignalOutput`:
      1. Extract close, high, low from `bar_arrays[config.primary_timeframe]`
      2. If primary timeframe not available or arrays empty: return neutral SignalOutput(module_name="timing", direction=0.0, confidence=0.0)
      3. Use last `config.ou_lookback_bars` bars for OU estimation
      4. OU estimation (via asyncio.to_thread since it involves numerical computation):
         - kappa, theta, sigma, ou_conf = await asyncio.to_thread(estimate_ou_parameters, close_prices[-lookback:])
      5. Entry window:
         - direction, urgency, window_conf = compute_entry_window(kappa, theta, sigma, close_prices[-1], ou_conf)
      6. Phase transition:
         - state, energy, phase_conf = detect_phase_transition(close, high, low, config.phase_transition_atr_period, config.phase_transition_compression_threshold, config.phase_transition_expansion_threshold)
      7. Combine timing signals:
         - If state == "compression" and energy > 0.3: boost urgency by energy * 0.5 (breakout imminent)
         - If state == "expansion": keep direction from OU, reduce confidence slightly (move already happening)
         - final_direction = direction (from OU model)
         - final_confidence = (window_conf * 0.6 + phase_conf * 0.4) * urgency -- low urgency near mean = low confidence
         - Clamp confidence to [0, 1]
      8. Return SignalOutput(
           module_name="timing",
           direction=np.clip(final_direction, -1, 1),
           confidence=final_confidence,
           metadata={"kappa": kappa, "theta": theta, "sigma": sigma, "half_life": np.log(2)/kappa if kappa > 0 else float('inf'), "phase_state": state, "energy": energy, "urgency": urgency}
         )
    - Per D-12: timing has NO veto or delay power. It just contributes to the blend.
